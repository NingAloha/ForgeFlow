# Git Workflow

## 目标
- `main`：稳定主线，只通过 PR 合并。
- 主题分支：并行推进，但避免长期分叉失控。

## 分支类型
- `track/*`：长期方向分支（控制面、实现治理、测试语义、TUI 观测、文档等）。
- `feature/*`：短期功能开发分支。
- `fix/*`：缺陷修复分支。
- `docs/*`：文档改动分支。
- `refactor/*`：结构调整分支。

## 默认流程
1. 从 `main` 拉取最新并创建工作分支。
2. 在工作分支完成改动并本地自检。
3. 发起 PR 到 `main`，等待检查通过后合并。
4. 合并后删除已完成分支。

## 收敛规则
- `track/*` 只保留少量活跃轨道，避免“平行宇宙”。
- `track/*` 完成一个里程碑就合并回 `main`。
- 连续两周无后续计划的 `track/*` 分支应删除。
- `feature/*` / `fix/*` / `docs/*` / `refactor/*` 默认短命，合并即删。

## 同步节奏
- 分支空闲时不必频繁同步 `main`。
- 开始开发前同步一次 `main`。
- 提交 PR 前再次同步一次 `main`，减少 out-of-date 冲突。

## Tag 规则
- 只在 `main` 的合并提交上打阶段性 tag。
- 不在 `track/*` 或短命分支上打正式版本 tag。

## 质量门禁
- 推荐本地执行：
  - `ruff check .`
  - `pytest -q`
- 以仓库实际 CI 结果作为最终合并依据。
