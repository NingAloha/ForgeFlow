Field meanings:
non_goals:
Explicitly stated product scope exclusions, unsupported areas, deferred capabilities, or things the product will not do.
Each non_goal item must include:
- kind: commitment strength of the non-goal
- text: concise non-goal statement
no_non_goals_declared:
true only when the user clearly says there are no explicit non-goals.
detected_inconsistencies:
Issues that prevent the answer from being treated as a completed non-goals clarification.
Boundary model:
non_goals captures negative product scope with commitment strength.
explicit_constraints captures mandatory rules the system must obey.
General negative scope statements belong to non_goals.
Prohibitions that function as policy/security/data/compliance constraints belong to explicit_constraints.
Allowed kind labels:
- permanent: the product should not support this in principle or long-term.
- release: the current version / MVP / first release will not support this.
- deferred: this is postponed or may be considered later.
Examples that belong to non_goals:
- 首版不做评价系统
- MVP 不做移动端
- 暂时不支持跨校交易，后续再考虑
- 原则上不支持校外用户交易
Examples that belong to explicit_constraints, not non_goals:
- 禁止公开学生手机号
- 禁止校外用户访问
- 禁止存储明文密码
- 禁止绕过学校统一认证
- 交易数据不得向第三方共享
Extraction rules:
- Extract only non-goals explicitly stated by the user's answer.
- Do not infer non-goals from product type.
- Do not invent non-goals.
- Do not include duplicate non-goals.
- Use concise, stable text.
- Preserve important scope details such as version boundaries, release boundaries, unsupported platforms, unsupported user groups, unsupported workflows, and excluded capabilities.
- If the user clearly states there are no explicit non-goals, set no_non_goals_declared to true and return an empty non_goals array.
- If no_non_goals_declared is true, non_goals must be empty.
- If non_goals is non-empty, no_non_goals_declared must be false.
- "暂无明确不做的范围" is a valid completion answer, not an inconsistency.
Commitment mapping rules:
- If the user says "首版不做 X", use kind="release".
- If the user says "MVP 不做 X", use kind="release".
- If the user says "暂时不做 X / 后续再考虑 X", use kind="deferred".
- If the user says "永远不做 X / 原则上不做 X / 长期不支持 X", use kind="permanent".
- Do not infer permanent exclusion from vague wording.
- If the user only says "不做 X" and the question context does not clearly scope it to current release/MVP, report ambiguous_non_goal_commitment.
- If the question itself asks about current version/MVP/current stage, then "不做 X" may be interpreted as kind="release".
Do not extract as non-goals:
1. Target users already captured by product.target_users.
Example:
"主要面向大学生"
Do not extract.
2. Application type already captured by product.application_type.
Example:
"做 Web 应用"
Do not extract.
3. Target platforms already captured by product.target_platforms.
Example:
"支持 Web"
Do not extract.
4. Capability categories already captured by scope.capability_categories.
Example:

Do not extract as non-goals.
5. Technical stack constraints.
Example:
"不用 Redis"

6. Policy/security/data/compliance prohibitions.
Example:
"禁止公开学生手机号"
This is closer to explicit_constraints. Do not extract as non-goals.
Inconsistency detection:
Report detected_inconsistencies only for issues directly related to scope.non_goals.
Use these inconsistency ids when applicable:
1. ambiguous_non_goal_commitment
Use when the answer says "不做 X" but does not clarify permanent/release/deferred commitment in current context.

2. vague_non_goal
Use when the answer contains vague exclusions that cannot guide later stages.
Example:

"别做太多"
"先简单点"
Message example:
"用户回答过于宽泛，无法形成可执行的非目标边界。"
3. uncertain_non_goals_absence
Use when the user gives a vague, uncertain, evasive, or placeholder answer that does not clearly declare whether non-goals exist.
Examples:

"用户没有明确声明是否存在不做或暂不支持的范围，需要进一步确认。"
4. functional_requirement_instead_of_non_goal
Use when the answer primarily lists features or capability items rather than non-goals.
Example:
"发布商品、搜索商品、私信沟通"
Message example:
"用户回答主要描述功能或能力，而不是明确不做或暂不支持的范围。"
5. explicit_constraint_instead_of_non_goal
