Field meanings:
explicit_constraints:
Additional user-stated constraints not already represented by first-class fields.
no_explicit_constraints_declared:
true only when the user clearly says there are no additional explicit constraints.
detected_inconsistencies:
Issues that prevent the answer from being treated as a completed explicit constraints clarification.
Constraint model:
ForgeFlow treats requirements capture as progressive constraint formation.
The following are first-class fields and should not be duplicated into explicit_constraints:
- product.target_users
- product.application_type
- product.target_platforms
- scope.capability_categories
- scope.non_goals
scope.explicit_constraints stores remaining additional constraints explicitly stated by the user.
Allowed constraint kind labels:
- "technical"
- "platform"
- "policy"
- "resource"
- "performance"
- "integration"
- "data"
- "business"
- "other"
Kind meanings:
technical:
Technical stack, implementation dependency, architecture-level technology limitation, library/database/framework restriction.
platform:
Platform, runtime, deployment target, or compatibility limitation not already captured by product.target_platforms.
policy:
Legal, school, organization, privacy, authentication, access-control, compliance, moderation, or governance constraint.
resource:
Time, budget, staffing, maintenance, operational capacity, team skill, or delivery resource constraint.
performance:
Scale, latency, throughput, storage size, upload size, concurrency, availability, or performance limitation.
integration:
External system, third-party service, payment, notification, identity provider, or API integration constraint.
data:
Data retention, data visibility, sensitive data, storage location, auditability, deletion, or data-sharing constraint.
business:
Business rule, transaction rule, trust rule, eligibility rule, workflow policy, or domain-specific operating constraint.
other:
Explicitly stated constraint that is clearly a constraint but does not fit the above kinds.
Extraction rules:
- Extract only constraints explicitly stated by the user's answer.
- Do not infer constraints from product type.
- Do not infer constraints from target users, application type, target platforms, or capability categories.
- Do not duplicate first-class fields into explicit_constraints.
- Do not invent constraints.
- Do not include duplicate constraints.
- Use concise, stable text.
- Preserve important details such as numbers, deadlines, technologies, organizations, platforms, and prohibitions.
- If the user clearly states there are no additional constraints, set no_explicit_constraints_declared to true and return an empty explicit_constraints array.
- no_explicit_constraints_declared must be true only when the user explicitly states that there are no additional constraints.
- Do not set no_explicit_constraints_declared to true for vague, uncertain, evasive, or placeholder answers.
- Examples that mean no additional constraints:
  - 暂无其他约束
  - 没有其他约束
  - 除了前面说的没有了
  - 暂时没有额外限制
  - 没有补充约束
- Examples that do NOT mean no additional constraints:
  - 先这样
  - 暂时没想好
  - 不确定
  - 应该没啥
  - 随便
  - 你看着办
- If no_explicit_constraints_declared is true, explicit_constraints must be empty.
- If explicit_constraints is non-empty, no_explicit_constraints_declared must be false.
- Technical stack constraints are valid explicit constraints.
- "必须用 React、PostgreSQL 和 Redis" is a valid technical constraint.
- "暂无其他约束" is a valid completion answer, not an inconsistency.
- Do not extract negative product scope exclusions as explicit constraints.
- Answers such as:
  - 首版不做 X
  - MVP 不做 X
  - 当前版本不支持 X
  - 暂不做 X
  - 后续再考虑 X
  - 原则上不做 X
  belong to `scope.non_goals`, not `scope.explicit_constraints`.
Do not extract as explicit constraints:
1. Target users already captured by product.target_users.
Example:
"主要面向大学生"
Do not extract.
2. Application type already captured by product.application_type.
Example:
"做 Web 应用"
Do not extract.
3. Target platforms already captured by product.target_platforms.
Example:
"支持 Web"
Do not extract.
4. Capability categories already captured by scope.capability_categories.
Example:
"发布商品、搜索商品、私信沟通"
Do not extract as explicit constraints.
5. Non-goals.
Example:
"不做移动端"
This is a non-goal and should not be extracted into explicit_constraints.
Inconsistency detection:
Report detected_inconsistencies only for issues directly related to scope.explicit_constraints.
Use these inconsistency ids when applicable:
1. vague_explicit_constraint
Use when the answer contains vague constraints that cannot guide later stages.
Example:
"做得好一点"
"尽量简单"
"性能要好"
"体验要好"
Message example:
"用户回答过于宽泛，无法形成可执行的明确约束。"
2. first_class_field_repetition
Use when the answer only repeats already captured first-class fields and does not provide additional explicit constraints.
Example:
"还是大学生，Web 应用，支持 Web"
Message example:
"用户回答重复了已结构化的一等字段，没有提供额外明确约束。"
3. functional_requirement_instead_of_constraint
Use when the answer primarily lists features or capability items rather than constraints.
Example:
"发布商品、搜索商品、私信沟通"
Message example:
"用户回答主要描述功能或能力，而不是额外明确约束。"
4. non_goal_instead_of_explicit_constraint
Use when the user's answer primarily describes non-goals rather than additional explicit constraints.
Example:
"首版不做社交功能"
Message example:
"用户回答更像产品非目标，而不是额外显式约束，需要在 non_goals 层澄清。"
If there is no clear inconsistency, return:
"detected_inconsistencies": []
