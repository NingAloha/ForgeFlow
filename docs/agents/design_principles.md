# Agent Design Principles

这一页记录当前项目在架构演进上准备吸收的 agent 设计原则。

它不是工作流规则文档，也不直接替代状态契约。它主要回答：

* 这套系统下一步为什么要这样演进
* 从外部成熟 agent 产品里，哪些经验值得吸收
* 这些经验落到 ForgeFlow 当前架构时，应如何解释

## 为什么要补这页

当前仓库已经把主流程、阶段判定、回流规则和状态契约拆清楚了，但还缺一层“设计哲学约束”。

少了这一层，后续很容易出现几类偏移：

* 一遇到新问题就加一个新 stage 或新状态文件
* 把 agent 能力增强误解成“往上下文里塞更多信息”
* 用越来越多的提示词和固定提醒弥补动作空间设计不足
* 把 orchestrator 做成硬编码流程机，而不是 agent 协作的控制层

因此这里先明确：ForgeFlow 后续不是朝“更多规则、更大 prompt、更复杂阶段图”演进，而是朝“更合适的动作空间、更稳定的结构化交互、更可控的上下文构建”演进。

## 当前应吸收的几条原则

## 1. 工具要贴合 agent 能力，而不是贴合功能清单

ForgeFlow 当前最核心的动作空间不是 shell 命令，而是：

* orchestrator 做阶段决策
* 每个 stage agent 负责本阶段产物
* 各阶段通过 `state/*.json` 交接

这意味着，后续设计动作时，优先考虑的问题不是“还缺哪几个功能”，而是：

* 当前 agent 是否容易理解什么时候该做这一步
* 当前 agent 是否容易正确地产出结构化结果
* 当前动作是否和现有阶段职责边界一致

对应到当前项目，优先扩展的通常不应是更多 stage，而应是 stage 内部更稳定的动作类型，例如：

* 结构化提问
* 状态检索
* 上下文补充读取
* 结构化任务拆解
* 契约校验

## 2. 能用结构化交互解决的，不要退回自由文本

用户澄清、阻塞问题、任务状态、回流原因，这些内容如果只存在于摘要文本里，系统后续几乎无法可靠消费。

因此 ForgeFlow 后续应优先把下面几类交互结构化：

* 需求阶段的未决问题
* 方案阶段的选择分歧
* 实现阶段的 blocker
* 测试阶段的问题归因

这条原则的含义不是“不允许自然语言”，而是：

* 自然语言可以作为解释层
* 但控制层决策必须尽量依赖结构化字段

换句话说，文本可以面向人，状态必须面向系统。

## 3. 上下文应该由 agent 逐步构建，而不是一次性灌满

当前 `AgentContext` 会把所有状态一次性交给 agent，这在 MVP 阶段可行，但长期会带来几个问题：

* 当前 stage 会被无关下游状态污染
* agent 更容易被旧状态误导
* TUI 和 orchestrator 会倾向于不断追加上下文，而不是管理信息边界

ForgeFlow 后续更合理的方向是渐进披露：

* 先给当前阶段最小必要上下文
* 需要更多信息时，再显式读取对应状态或文档
* 把“能否读取更多上下文”视为能力设计的一部分

这和当前“契约驱动”的思路是相容的，因为契约本来就强调：

* 上游只交付本阶段真正需要的结果
* 下游按需消费，而不是默认获得一切

## 4. 旧护栏要定期退役，否则会变成上限

很多早期系统都会先依赖重提醒、硬编码步骤、固定 checklist 来稳住行为。

这在 ForgeFlow 里也可能出现，例如：

* 某些阶段判定规则写得过于僵硬
* agent 被要求机械地产出固定格式摘要
* orchestrator 过早接管了本该由 agent 自主判断的细节

这些护栏在早期是必要的，但模型和系统能力提高后，需要定期复盘：

* 哪些规则仍然在防错
* 哪些规则已经开始限制 agent 调整路径
* 哪些提醒应该被替换为更好的状态结构或任务结构

原则上，应优先通过更好的状态设计和动作设计减轻约束，而不是无限叠加提醒。

## 5. 扩能力不一定扩工具，也可以扩“发现路径”

ForgeFlow 不必把每种能力都做成一个新的 agent 或一个新的阶段。

更合理的扩展路径通常有三类：

* 增加某阶段可读取的结构化文档
* 增加某阶段可使用的技能性说明
* 增加 orchestrator 和 agent 之间共享的中间状态

这条原则的核心是控制复杂度：

* 新增一个阶段，会改变状态机
* 新增一个全局工具，会扩大动作空间
* 新增一个按需发现路径，通常只增加局部复杂度

因此在没有充分证据前，不要把局部问题上升为新的主流程节点。

## 这些原则和当前架构的对应关系

当前代码结构里，最适合承接这些原则的是以下几层：

* [../../agents/orchestrator/](../../agents/orchestrator/)：继续负责阶段真相、前进、回流与执行入口，但不要吞掉本该结构化表达的问题。
* [../agents/base.py](../agents/base.py)：后续可扩展更明确的 action/result 类型，而不只是单一 `AgentResult`。
* [../state/contracts.md](../state/contracts.md)：需要承接新的结构化状态，例如问题、任务、上下文读取结果。
* `tui/`：后续负责把结构化问题、任务状态和阶段进度呈现给用户，而不是只显示日志文本。

## 当前文档层面的落地约束

在真正开始代码实现前，先约束几条文档原则：

* 不把“新增 stage”当成默认扩展手段。
* 不把“大 prompt + 更多提醒”当成默认稳定手段。
* 优先新增结构化状态，而不是新增自由文本摘要要求。
* 优先让 agent 按需构建上下文，而不是默认广播全部状态。
* orchestrator 优先做控制与决策，不承担所有知识和所有交互。

## 建议优先演进顺序

如果按投入产出比排序，当前最值得优先推进的是：

1. 结构化提问与回答
2. 结构化任务对象，而不是阶段内散落备注
3. 渐进式上下文加载

原因很直接：

* 结构化提问先解决“需求和方案阶段怎么把信息问清楚”
* 结构化任务再解决“实现和测试阶段怎么持续跟踪工作对象”
* 渐进式上下文最后解决“随着系统变复杂，如何控制噪音和误导”
