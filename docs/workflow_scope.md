# Workflow Scope

这一页定义当前这套工作流规则的适用场景与边界，避免把它误解为已经覆盖所有软件开发协作模式。

## 当前适用场景

当前文档中的工作流，主要适用于以下场景：

* 一个全新的项目，或一次新的、边界相对独立的交付轮次。
* 用户能够在流程前期把“本轮要解决的问题”收敛到清晰需求，而不是长期悬而未决。
* 当前工作可以围绕一个主线闭环推进，即从 requirements 一路推进到 testing / done。
* 当前主要是单线程推进，而不是多个需求流、多个版本线或多个模块并行演进。
* 当前希望由 orchestrator 基于结构化状态做阶段切换，而不是高度依赖人工临场判断。

更具体地说，这套规则最适合：

* 从 0 到 1 的新项目梳理与落地。
* 一个 feature 边界比较完整、可以单独定义需求和验收标准的开发轮次。
* 先收敛需求，再逐层推进方案、设计、实现和验证的工作方式。

## 当前工作流的核心假设

当前规则建立在以下假设之上：

* 当前只有一条主流程：`INIT -> REQUIREMENTS -> SOLUTION -> DESIGN -> IMPLEMENTATION -> TESTING -> DONE`。
* 当前只要求“本轮需求闭环”清晰，不要求一次性定义项目未来所有需求。
* 回流存在，但回流服务于修正本轮闭环中的问题，而不是长期并行管理多个变化源。
* `state/*.json` 当前表达的是“这一轮主流程的当前状态”，而不是多轮并存、多分支并存的状态集合。
* 当用户提出新的明确需求时，更接近开启下一轮流程，而不是无限并入当前 `DONE` 或当前活动轮次。

## 当前不重点覆盖的场景

以下场景目前不属于这套规则的重点覆盖范围，或至少尚未被完整建模：

* 一个长期演进中的存量项目，持续接收零散需求、缺陷修复和架构调整。
* 多个 feature 并行推进，且彼此有不同节奏、不同阶段状态。
* 用户在实现中持续插入新需求，导致当前轮次目标不断变化。
* 同时维护多个版本线、发布线或环境差异较大的交付流。
* 一套状态文件同时管理多个子任务、多个模块独立进度或多个需求批次。
* 以 issue backlog、sprint、迭代计划为核心的持续开发模式。

## 当前边界下的解释原则

为了避免误用，当前应按以下方式理解这套工作流：

* 它不是通用项目管理系统，而是“单轮需求闭环驱动”的工程工作流。
* 它不是为了替代 backlog、roadmap 或多版本协同机制。
* 它不要求用户一次说完整个项目的全部未来需求，但要求本轮目标能尽早收敛。
* 如果新的需求已经改变当前轮次目标，应优先判断是否开启新一轮，而不是强行并入当前阶段。
* 如果未来要支持增量开发、多轮并行或长期演进，应在状态模型和 orchestrator 规则上显式扩展，而不是默认复用当前单轮模型。

## 与后续演进原则的关系

当前边界并不意味着系统未来只能停留在“单轮主流程”。

但后续扩展应遵守几个约束：

* 不因为出现新交互需求，就立刻新增主流程阶段。
* 不因为信息不足，就默认把更多上下文一次性塞给所有 agent。
* 不因为流程暂时不稳，就持续叠加硬编码提醒和文本约束。

更合理的扩展顺序应是：

* 先补结构化交互，例如问题、回答、任务、阻塞项。
* 再补按需上下文发现能力。
* 最后再判断是否真的需要扩展主状态机。
